Phase 7 — Slovak eHealth Analysis

Pochopiť ako slovenské zdravotníctvo funguje dnes. Cieľom nie je navrhnúť zmenu — ale vedieť čítať, interpretovať a preložiť eZdravie do FHIR a EHDS.
Strategická hodnota: Väčšina FHIR konzultantov vie FHIR. Málokto vie zároveň eZdravie, NCZI IM, XML/XSD/SOAP a vedieť to preložiť do FHIR a EHDS. Toto je vzácna kompetencia špecifická pre slovenský kontext.

Moduly na analýzu

ModulPrečo prioritný
eLab v3 Najštruktúrovanejší, najčistejší dátový model. Mapovanie na Observation + DiagnosticReport je priamočiare.
eVyšetrenie v8/v9 Najklinicky bohatší modul — ambulantný nález. Komplexné sekcie, voliteľné elementy, diagnózy, výkony.
JRUZ — priečna kapitola (nie samostatný modul): Rodné číslo, JRÚZ ID, poistný kód, IČO PZS, kód lekára sú referencované v oboch moduloch. Analyzujú sa ako súčasť eLab a eVyšetrenie analýzy.

Phase 9 — Architecture Patterns

Rozhodnúť ako sa FHIR integruje na systémovej úrovni. Architektonické rozhodnutia sú nadradené technologickým — Facade vs Hybrid vs FHIR-native určuje celý trajectory projektu, náklady a timeline.
Prečo Architecture pred Profiles: Profily sú technický výraz architektonického rozhodnutia. Napríklad FhirSkPatient.identifier cardinality závisí od toho, či buduješ Facade (JRUZ vždy prítomný v kontexte) alebo standalone systém. Profil napísaný pred architektúrou je dohadom.

Core decision framework (Darren Devitt)

Facade
FHIR REST API pred existujúcim SOAP/XML systémom. Bez migrácie, rýchle nasadenie, obmedzená FHIR hĺbka. Slovenský prípad: eZdravie SOAP → FHIR Facade → EHDS endpoint.
Risk: FHIR je tenká vrstva, legacy obmedzenia presakujú
Hybrid ✓ SK 2025–2030
Nové FHIR moduly vedľa legacy systémov + integračná vrstva. Niektoré dáta FHIR-native (nové moduly), niektoré Facade (legacy eZdravie).
Najrealistickejší scenár pre slovenské zdravotníctvo
FHIR-native
Greenfield. Nové systémy postavené na FHIR R4 od začiatku. Maximálna FHIR hĺbka, najvyššia migračná cena z legacy.
Target state pre EHDS 2027+ nové implementácie

Témy

TémaČo sa naučíš
Facade vs Hybrid vs FHIR-nativeCost/risk tradeoffs, kedy čo použiť, decision checklist
FHIR Facade over eZdravieWrapping SOAP/XML za FHIR REST — konkrétny SK vzor
FHIR SubscriptionsEvent-driven notifikácie — R4 vs R5 model
SMART on FHIROAuth2 pre FHIR — EHR launch, standalone, backend services
Bulk Data ($export)NDJSON async export, EHDS sekundárne využitie
HAPI FHIR cloudRailway deploy, hapi.fhir.sk, production konfigurácia
Slovenský kontext: Každé PZS s EHDS deadline 2027 bude čeliť presne tomuto rozhodnutiu: Facade eZdravie, Hybrid, alebo replace? Porozumenie architektonickým možnostiam a tradeoffom je jadro consulting hodnoty.

Success criteria

  • Facade vs Hybrid vs FHIR-native decision framework zdokumentovaný s konkrétnym SK príkladom
  • FHIR Facade nad eZdravie SOAP/XML — implementačný vzor
  • SMART on FHIR auth flow — EHR launch + standalone
  • Bulk Data $export — async pattern, NDJSON
  • HAPI FHIR nasadený na Railway (hapi.fhir.sk)

Phase 10 — Slovak FHIR Profiles

Vytvoriť Slovak FHIR profily odvodené z Phase 8 mapovania a Phase 9 architektonických rozhodnutí. Každý constraint má dôvod zdokumentovaný v Phase 7–9. Žiadna profile práca bez dokončeného mapovania a bez jasného architektonického kontextu.
Prečo nie skôr: Súčasný FhirSkPatient v0.2.0 bol napísaný bez analýzy zdrojového systému a bez architektonického kontextu — je to dohadom. Phase 10 ho nahradí profilmi ktoré vychádzajú z reality.

Phase 11 — Implementation Guide + Terminology Architecture

Dva paralelné tracky — publikovať Slovak FHIR profily ako formálny IG, a postaviť terminologickú infraštruktúru. Pre EHDS sú oba povinné.

Track A — Slovak FHIR Implementation Guide

FSH files
→
SUSHI compiler
→
StructureDefinition JSON
→
HL7 IG Publisher
→
HTML IG site

FSH (FHIR Shorthand) — domain-specific jazyk pre písanie FHIR profilov. Nahrádza 200-riadkový StructureDefinition JSON čitateľnými constraintami. Priemyselný štandard pre IG authoring.

IG štruktúra: Home, Profiles (differential + snapshot + examples), Terminology (ValueSets, CodeSystems), Examples, Downloads. Output: Slovak FHIR IG — minimum lokálne, cieľ: fhir.sk subdoména alebo GitHub Pages.

Track B — Terminology Architecture

ComponentToolÚčel
SNOMED CT Snowstorm (IHTSDO) Hierarchia, ECL queries, $expand — purpose-built, nie HAPI FHIR
LOINC HAPI FHIR + LOINC / Ontoserver Lab code lookup, panel definitions
Custom CodeSystems HAPI FHIR Slovak číselníky: špecializácie, kódy zariadení

Terminology governance Process Versioning, deprecation, ownership — nie len tooling
